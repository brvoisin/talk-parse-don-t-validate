La fonction est partielle :
toutes les entrées possibles ne sont pas prises en compte.

On démarre avec une fonction toute simple qui retourne le premier élément d’une séquence. Mais si on regarde bien… qu’est-ce qui pourrait mal tourner ? …

⏭️ 👉 Une séquence vide, évidemment. Et là : erreur.

⏭️ Le souci, c’est que cette fonction est partielle. Elle ne gère pas tous les cas possibles. Ce qu’on veut, c’est la rendre totale. Autrement dit : qu’elle fonctionne pour toutes les entrées, sans planter. [Répétition]

Fonction totale

Toutes les entrées possibles sont prises en compte.

Il y a 2 solutions pour rendre notre fonction totale.

À l’inverse d’une fonction partielle, une fonction totale gère tous les cas possibles, sans planter. Alors, comment on transforme notre fonction pour qu’elle le soit ?

⏭️ 👉 Il y a deux options. Et on commence tout de suite par celle qui semble la plus évidente.

Solution 1 : gérer la séquence vide


            from collections.abc import Sequence

            def head[T](seq: Sequence[T]) -> T | None:
                if not seq:
                    return None
                return seq[0]
          

              print(head([]))
              print(head([1, 2, 3]))
            

              None
              1

Première solution : on gère le cas de la séquence vide. La fonction renvoie soit un élément, soit None si la séquence est vide.

⏭️ 👉 En l’exécutant, plus d’erreur : une liste vide donne None, tout simplement.

Maintenant, on va voir ce que ça donne dans un exemple concret.

Solution 1 : gérer la séquence vide


            from collections.abc import Sequence
            from typing import NewType

            NothingType = NewType("NothingType", str)
            Nothing = NothingType("Nothing")

            def head[T](seq: Sequence[T]) -> T | NothingType:
                if not seq:
                    return Nothing
                return seq[0]
          

              print(head([]))
              print(head([1, 2, 3]))
              print(head([None]))

              1

Premier truc qui saute aux yeux : ⏭️ on se retrouve à vérifier la valeur de retour de head partout. C’est répétitif, un peu lourd… et dans des cas plus complexes — genre dans une boucle — ça peut même impacter les perfs.

⏭️ Mais surtout, le vrai problème, c’est que ça ouvre la porte à des bugs. Et on va voir pourquoi.

            import os
            from pathlib import Path
            from typing import TypeGuard, TypeVar, Unpack

            T = TypeVar("T")
            NonEmptyTuple = tuple[T, Unpack[tuple[T, ...]]]


            def head(seq: NonEmptyTuple[T]) -> T:
                return seq[0]


            def get_config_directories() -> NonEmptyTuple[Path]:
                config_dirs_string = os.getenv("CONFIG_DIRS", "")
                config_dirs = tuple(
                    Path(dir)
                    for dir in config_dirs_string.split(",")
                    if dir
                )
                if not is_non_empty(config_dirs):
                    raise ValueError("CONFIG_DIRS cannot be empty")
                return config_dirs


            def is_non_empty(
                seq: tuple[T, ...]
            ) -> TypeGuard[NonEmptyTuple[T]]:
                return len(seq) > 0


            def main() -> None:
                config_dirs = get_config_directories()
                initialize_cache(head(config_dirs))


            def initialize_cache(cache_dir: Path) -> None: ...


            if __name__ == "__main__":
                main()

Quel est le rapport ?

Parse, don’t validate

Envoi d’un message à un numéro de téléphone


            def validate_phone_number(text: str) -> bool:
                return len(text) == 10 and text.isdigit()


            def send_message(phone_number: str, message: str) -> None:
                if not validate_phone_number(phone_number):
                    raise ValueError("Invalid phone number")
                print(f"Sending message to {phone_number}: {message}")

Le numéro de téléphone est une chaîne de caractères, par précaution on vérifie son format à chaque envoi.

Parse, don’t validate

Envoi d’un message à un numéro de téléphone


            from typing import NewType

            ValidPhoneNumber = NewType("ValidPhoneNumber", str)


            def parse_phone_number(text: str) -> ValidPhoneNumber:
                if len(text) != 10 and not text.isdigit():
                    raise ValueError("Invalid phone number")
                return ValidPhoneNumber(text)


            def send_message(
                phone_number: ValidPhoneNumber, message: str
            ) -> None:
                print(f"Sending message to {phone_number}: {message}")

Le type indique que le numéro est valide, plus besoin de vérifier son format à chaque envoi.

Parse, don’t validate

  • Validez qu’une fois vos données à l'entrée
  • Conservez l’information avec des types
    (stop à la primitive obsession)
  • Des bugs évités grâce au typage avant même d’exécuter le code
    (IDE, compilation ou analyse statique de code)

Pour conclure, si vous devez retenir une chose : ⏭️ validez vos données une seule fois, à l’entrée. C’est simple, mais redoutablement efficace. Ça vous évitera bien des erreurs plus loin dans le traitement — surtout si des effets de bord ont déjà commencé à faire leur vie avant que ça plante.

⏭️ Ensuite, typez vos données. Pas juste pour faire joli mais parce que ça les rend plus claires, plus sûres, et surtout, ça aide tout le monde à comprendre ce qu’on manipule. Créez des types qui parlent vraiment de votre métier. Ne vous contentez pas des types primitifs.

⏭️ Le résultat ? Moins de bugs, des erreurs détectées avant même d’exécuter le code, et un IDE ou un compilateur qui vous dit tout de suite : “Hé, là y’a un truc qui colle pas.” C’est ça, le vrai gain : un code plus fiable, plus lisible, et une équipe qui travaille plus sereinement.
